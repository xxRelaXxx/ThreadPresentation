What is a Thread?

A thread is the smallest unit of execution within a process. Think of it as a separate "path" of execution.

While a Process is a full program (like opening Word), a Thread is a specific task happening inside it (like typing while the spell-checker runs).

Why Do We Use Threads?

Parallel Tasks

Execute multiple operations simultaneously, utilizing modern multi-core CPUs efficiently.

Client Handling

Servers need to handle hundreds of users at once. One thread per client ensures no one waits in line.

Background Work

Perform heavy calculations, file I/O, or database queries without freezing the main application.

Responsiveness

Keep the UI (User Interface) reactive even while heavy processing happens in the background.

1

Extending Thread (Concept)

  • Class IS a thread: Your class inherits directly from the Thread class.
  • Override run(): You must implement the run() method, which contains the logic that will be executed by the new thread.
  • Call .start(): This is the key command that tells the JVM to launch a new thread and call the run() method asynchronously.
  • Simple but not flexible: It's straightforward, but Java's single inheritance rule prevents the class from extending anything else.

When to Extend Thread?

Simple Cases

Best for small, isolated projects or quick scripts where architectural constraints are minimal.

Thread Is the Object

When the concept of the object itself is inherently a thread (e.g., a simple custom scheduler).

Avoid in Big Systems

Because it uses up your single chance at inheritance, it's not scalable for complex architectures.

2

Implementing Runnable (Concept)

Focus on Separation of Concerns

Runnable Interface

Represents the TASK LOGIC. It defines what needs to be executed concurrently.

public void run() { ... }

Thread Class

Represents the EXECUTOR. It defines how to run the task using system resources.

new Thread(runnable).start()

3

ExecutorService (Concept)

The modern solution for managing concurrency using Thread Pools.

  • Thread Pools: It manages a set number of worker threads, avoiding the overhead of creating new ones constantly.
  • Avoid Manual Creation: You submit the task; the ExecutorService handles the thread mechanics.
  • Ideal for Servers: Essential for handling concurrent requests efficiently and preventing system resource exhaustion.
  • Automatic Scheduling: Tasks are queued and automatically scheduled when a thread becomes available.

ExecutorService Advantages

Speed

Reusing threads is much faster than creating and destroying a new thread for every single task.

Reusability

The underlying worker threads are recycled, reducing resource consumption and improving throughput.

Scalability

Limits resource usage and ensures the application doesn't overload the server with too many concurrent threads.

Cleaner Code

Abstracts away the manual thread management, leading to simpler, more readable concurrency code.

The Java Thread Lifecycle

A thread moves through several defined states from creation until death.

NEW

Thread object has been created but not yet started (new Thread()).

RUNNABLE

Thread is ready to run and waiting for the OS scheduler (after .start()).

RUNNING

The thread is actively executing its run() method on the CPU.

BLOCKED/WAITING

Thread is temporarily paused (e.g., waiting for a lock, .sleep(), or .join()).

TERMINATED

The thread has finished execution (run() method completed or an exception occurred).

The methods we cover next are what control these transitions.

Essential Thread Control Methods

These methods allow you to manage a thread's state, priority, and interactions.

.start()
.run()
.sleep()
.isAlive()
.join()
.setDaemon()
.interrupt()
.stop() (Deprecated)

Now let's see each one quickly.

.isAlive()

What it does:

Checks if the thread has been started and has not yet died. It returns true if the thread is in the Runnable or Blocked/Waiting state.

Example:

Thread t = new Thread(() -> { /* Task logic here */ });
t.start();
System.out.println(t.isAlive()); // true (usually)

Use Case:

Checking when a background process (like a file download or heavy calculation) is still actively running.

.join()

What it does:

Causes the current thread (e.g., the main thread) to wait until the thread it is called on (e.g., t1) finishes execution.

Mini-Example:

t1.start();
t1.join(); // <-- Main thread waits here
System.out.println("t1 completed"); // Only prints after t1 is done

Use Case:

Ensuring that a critical preprocessing step, like initializing a resource or waiting for a client to finish sending its setup data, completes before the program proceeds.

.interrupt()

What it does:

Sets the thread's interrupt status flag to true. It's a polite request to the thread to stop what it's doing.

Important:

It does **NOT** force the thread to stop. The thread must check the flag (Thread.interrupted()) or catch an InterruptedException (if sleeping/waiting) to terminate gracefully.

Use Case:

Cancelling long-running loops or calculations, or safely stopping a thread stuck in a sleep() or wait() state.

.setDaemon(true)

What it does:

Turns a thread into a background thread (or Daemon thread). The JVM won't wait for these threads to finish.

The Rule:

The entire Java application terminates when all **non-daemon** (user) threads have finished execution.

Use Case:

Ideal for non-critical, continuous background tasks like logging, garbage collection, or heartbeat monitoring.

Tie-in to Example 1: In FirstExample, setting c2.setDaemon(true) would make c2 stop instantly when the main thread finishes.

.stop() (Deprecated)

Why You Must NOT Use It:

  • Kills Threads Instantly: It forces the thread to stop without giving it a chance to clean up.
  • Data Corruption: Leaves any shared resources (like database connections or object locks) in an unpredictable and corrupted state.
  • Unsafe: Leads to application instability and difficult-to-debug errors.

Mentioned only because it exists no one uses it in modern, safe Java code.

Summary of Thread Controls

Method What it Does Safety/Use
.isAlive() Checks if thread is currently running or runnable. Safe
.join() Makes the calling thread wait until this thread finishes. Safe
.interrupt() Asks the thread politely to stop (sets a flag). Needs thread cooperation
.setDaemon() Turns thread into a background cleanup service. Safe
.stop() Kills the thread instantly. Dead, Never Use

The Real-World Message

In almost all modern, high-performance Java applications (especially servers), the preferred pattern is:

Runnable + ExecutorService

(This, combined with newer concepts like Java's Virtual Threads, defines how scalable concurrency is built.)
